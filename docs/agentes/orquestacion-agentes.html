Audita RLS, permisos, endpoints sin auth, variables expuestas y datos sensibles. Especialista en Supabase security.
No parchea. Audita y reporta vulnerabilidades.

Detecta queries lentas, renders innecesarios y problemas de escalabilidad. Solo actúa cuando el feature ya es correcto.

// prompt de orquestación completa ↑ click para copiar
Actúa como CTOAgent. Analiza esta funcionalidad: [DESCRIBE LA FUNCIONALIDAD] No programes todavía. Genera: 1. Arquitectura — Componentes del sistema — Flujos de datos — Dependencias con el sistema existente 2. Riesgos — Técnicos — De producto — De escalabilidad 3. Diseño de BD requerido — Tablas necesarias — Relaciones clave — Consideraciones de RLS 4. UX requerida — Pantallas necesarias — Flujos de usuario — Estados críticos 5. Plan de implementación — Fase 1: DataArchitectAgent → diseño de BD — Fase 2: UXUIAgent → wireframes y flujos — Fase 3: BackendAgent → APIs y migraciones — Fase 4: FrontendAgent → componentes y páginas — Fase 5: QAAgent → auditoría de bugs — Fase 6: SecurityAgent → revisión de permisos y RLS Coordina los agentes necesarios. Entrega el trabajo por fases. No empieces a programar hasta que apruebe el diseño.
// por qué funciona
01
Diseño antes que código
El costo de cambiar un diseño es cero. El costo de reescribir código mal diseñado es enorme. La aprobación explícita es el gate de calidad.
02
Especialización real
Un agente enfocado en seguridad piensa en seguridad. Uno enfocado en BD piensa en índices. La mezcla de roles produce soluciones mediocres.
03
Orden importa
BD primero, luego APIs, luego UI. Construir el frontend sobre APIs inexistentes produce deuda técnica inmediata y mocks que nunca se reemplazan.
04
Auditoría separada
El agente que construye no es el que audita. QA y Security como fases independientes, no mezcladas con la construcción, producen revisiones más honestas.
// cuándo saltar fases
Fix rápido (<30 min) — saltar todo, ir directo al agente correcto con contexto mínimo
Exploración / prototipo — solo CTOAgent + FullStackAgent para validar la idea rápido
Feature nuevo en sistema grande — el flujo completo siempre, sin atajos
Bug de producción — QAAgent → SecurityAgent → fix directo, sin pasar por diseño
// reglas de oro
✗
Nunca programar sin diseño aprobado
Si el CTOAgent entrega una arquitectura y el usuario no la aprueba explícitamente, ningún otro agente debe producir código. Esta es la regla más importante del framework.
✗
Nunca mezclar roles en un prompt
Pedirle al BackendAgent que también diseñe la UX produce peores resultados en ambas dimensiones. Un prompt, un rol, un output específico.
✗
No saltarse QA y Security antes de deploy
Son los únicos pasos no negociables. El costo de un bug de seguridad en producción supera cualquier ahorro de tiempo en el proceso.
✓
Pasar el output entre agentes
El output del CTOAgent va al DataArchitectAgent. El de Data va al Backend. El contexto acumulado entre fases es lo que produce coherencia en el sistema final.
